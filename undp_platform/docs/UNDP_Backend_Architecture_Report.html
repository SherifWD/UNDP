Objective

This document explains the backend architecture by first presenting the main system idea through architecture graphs, then expanding that view into a detailed explanation of the security model, workflow engine, storage design, accountability controls, reporting model, and production hosting plan.

The report is intended to show how the UNDP Platform backend supports secure municipal reporting, offline-capable mobile submission, validation workflows, donor-safe reporting, bilingual operation, and long-term operational accountability.

1. Architectural Overview

The backend is designed as a secure, role-aware, API-first Laravel system supporting offline-capable mobile reporting, role-based validation workflows, municipal and national aggregation, immutable auditability, donor-grade reporting, and bilingual Arabic / English operation.

1.1 Architectural Goals

  • Support field reporting from mobile clients that may operate with delayed connectivity.
  • Enforce least-privilege RBAC entirely at the API layer.
  • Separate compute, transactional data, and evidence media responsibilities.
  • Maintain a full immutable governance trail for operationally sensitive actions.
  • Provide municipality-scoped and nationally aggregated reporting views without leaking unauthorized raw data.

1.2 High-Level Architecture Principles

  • API-first, RESTful JSON contracts.
  • Strict RBAC enforcement at the API boundary and business-rule layers.
  • Stateless authentication for mobile and tokenized access for API clients.
  • Offline-tolerant submission lifecycle with idempotent retries.
  • Immutable audit logging for governance-critical actions.
  • Compute on VPS, large binary evidence in private object storage.

3. Authentication & Authorization (AUTH & RBAC)

3.1 Authentication Strategy

The platform uses Laravel Sanctum as the unified authentication layer. The current repository implements bearer-style access tokens and refresh tokens for API clients, including the Vue dashboard and mobile-facing flows, while keeping the backend compatible with first-party web deployment patterns.

ChannelMechanismCurrent BehaviorNotes
Web DashboardSanctum-protected API accessThe current Vue client stores a bearer token and sends it on protected requests.Used by admins, municipal users, auditors, and partner viewers through the web interface.
Mobile ClientSanctum personal access token + refresh tokenLogin returns access and refresh tokens with expiry metadata.Designed for secure device storage and app-managed token refresh.
Login MethodOTP-based identity verificationOTP requested and verified through Resala-backed endpoints.Localized responses and auditable auth events are built in.
Device Notification BindingFCM token registrationAuthenticated users can update device tokens for push notifications.Used after login to support workflow notifications.
Disabled User HandlingActive-user middlewareCurrent access token is revoked immediately if a disabled user attempts access.Prevents stale token use after account deactivation.

Concrete authentication controls in the current backend

  • OTP expiry: 300 seconds.
  • OTP resend cooldown: 60 seconds.
  • OTP max verification attempts: 5.
  • OTP code length: 6 digits.
  • OTP rate limiting: 6 requests per minute by IP and 5 requests per minute by phone number.
  • General API throttle: 240 requests per minute by user or IP.
  • Localized error messages returned in Arabic or English depending on resolved locale.

Security controls

  • Password or secret hashing uses Laravel hashing facilities. The current environment template exposes BCRYPT_ROUNDS=12; Argon2id remains a valid production policy alternative.
  • No credential material is hard-coded into the client application.
  • Refresh token rotation is enforced by deleting the old refresh token when a new pair is issued.
  • User disable events cause immediate token invalidation at middleware level and are written to the audit trail.

3.2 Role-Based Access Control (RBAC)

Access follows a permission-driven least-privilege model: access is explicitly granted, never implied. Client-side checks exist only to improve UX; all effective protection is performed on the server.

Key RBAC principles

  • Least privilege: each role receives only the permissions required to perform its duties.
  • Scoped permissions: visibility is constrained by municipality, ownership, and workflow state.
  • Server-side enforcement only: the backend remains the trust boundary.
  • No implicit access: absence of permission equals denial.

Representative permission patterns

  • submissions.view.own, submissions.view.municipality, submissions.view.all
  • submissions.validate, submissions.approve, submissions.reject, submissions.rework
  • projects.view, projects.manage
  • audit.view, workflow.manage
  • dashboards.view.system, dashboards.view.municipality, dashboards.view.partner
  • media.upload, reports.export.csv, reports.export.pdf

Supported role model

RoleMain PermissionsVisibility Model
Community ReporterCreate submissions, view own submissions, upload permitted media.No cross-municipality visibility. Own records only.
Municipal Validation RoleReview, validate, reject, and rework municipality-scoped submissions.Scoped to assigned municipality only.
Municipal Focal PointMunicipality worklists, monitoring, validation oversight, municipal dashboards.Full municipality visibility, no global dataset access.

Implementation note: the current repository exposes the concrete roles reporter, municipal_focal_point, undp_admin, partner_donor_viewer, and auditor. Municipality-scoped validation authority is enforced through permissions and the submission access layer.

3.3 Technological Enforcement (How)

RBAC is enforced through three mandatory backend layers working together. No API endpoint should return data outside the caller’s effective scope.

Query Scoping LayerEnsure data isolation in list and aggregate queries.SubmissionAccessService and ProjectAccessService constrain results by role, municipality, ownership, and approved-only visibility.

4. Submission Lifecycle & Workflow Engine

4.1 Submission States

StateMeaning
DraftLocally saved or server-side editable draft, primarily aligned with the mobile reporting flow.
QueuedPending synchronization or queued for further processing.
SubmittedReporter has formally submitted the payload to the backend.
Under ReviewVisible in validation worklists and pending municipality or admin action.
ApprovedAccepted into the official dataset and eligible for approved-only reporting.

Status and reason dictionaries are stored in dedicated workflow tables and settings structures, so the system can evolve operationally without requiring code deployment for every workflow label change.

Idempotency and retry safety

  • The mobile submission flow accepts a client_uuid and reuses existing submissions when the same identifier is submitted again.
  • Presigned media creation also supports a client-side media identifier to safely reuse an existing media asset record.
  • Retries should not create duplicate submissions or duplicate media records when client identifiers are reused correctly.

Workflow operating characteristics

  • Validation actions are explicit: approve, reject, and rework.
  • Status changes are recorded as timeline events.
  • Review decisions generate audit entries and can trigger push notifications.
  • Municipality and role scope continue to apply throughout the full submission lifecycle.

5. Media Handling & Evidence Management

5.1 Media Upload Design

Media is uploaded through dedicated authenticated endpoints. When the active media disk is S3-compatible, the backend creates a media metadata record, returns a time-limited pre-signed upload URL, and later marks the media as uploaded before queueing processing. The database stores metadata only, while binary files remain in object storage.

Design ElementCurrent / Target Behavior
AuthorizationUpload endpoints require authenticated access and permission checks. Submission visibility rules still apply.
ValidationServer validates type, size, count, and submission ownership or visibility before upload metadata is created.
Binary StorageFiles are stored in private object storage, not on the VPS disk.
Metadata PersistenceDatabase stores object key, media type, original filename, size, checksum, status, and variant references.
ProcessingCompletion dispatches a background job onto the media queue and updates status from pending to uploaded to processing to ready.

5.2 Compression Strategy

Media compression follows a hybrid client-plus-backend approach: compress as early as possible on the client to reduce field bandwidth and upload cost, then use background processing on the backend to normalize variants and delivery artifacts.

Operational processing note

The current repository already models media processing asynchronously through ProcessMediaAssetJob. In production, this queue boundary is the correct location to attach real image resizing, thumbnail generation, video transcoding, and poster extraction workers.

5.3 Secure transfer controls

  • Pre-signed upload URLs are time-limited. Current default upload TTL: 900 seconds.
  • Pre-signed download URLs are time-limited. Current default download TTL: 600 seconds.
  • Object storage remains private; no public-read policy is required for evidence.
  • Donor-facing users are prevented from directly receiving raw media URLs in restricted contexts.

6. Audit Logging & Accountability

6.1 Audit Log Coverage

The backend logs all sensitive and governance-critical actions, including authentication events, user status changes, role and permission-sensitive actions, submission status transitions, media operations, export events, and settings changes.

Audit AreaExamples
AuthenticationOTP requested, login success, token refreshed, token revoked for disabled user.
User governanceUser creation, profile update, role change, status disable or enable actions.
Submission workflowSubmission created, re-submitted, approved, rejected, or returned for rework.
Media operationsPre-signed upload issued, upload completed, processing ready, download URL issued.
ConfigurationGlobal settings updated, workflow or security settings changed.
ReportingExport task creation and report access events.

6.2 Audit Log Design

FieldDescription
TimestampStored as backend-created event time in UTC-capable datetime handling.
ActorUser identifier and associated role context when available.
ActionMachine-readable action code such as login success, media upload completed, or settings updated.
TargetEntity type and entity ID.

The audit model is append-only. In the current codebase, attempts to update or delete audit records throw a logic exception. Audit entries are therefore read-only through normal application behavior.

7. Web Validation & Oversight Modules

7.1 Validator Worklists

  • Validator and municipality-scoped worklists return only submissions that fall within the user’s allowed municipality scope.
  • Filtering is indexed and aligned to project, date, and status dimensions.
  • Default sort order favors the most recent operationally relevant records.
  • Shared endpoints remain safe because visibility is rechecked by policy and query scope, not only by route selection.

7.2 Validation Actions

  • Validation actions are explicit: approve, reject, or request rework.
  • Comments can be mandated by workflow policy for rejection or rework.
  • Updates should be treated transactionally: status transition, status event append, audit log write, and notification trigger belong to one coherent action boundary.
  • UNDP Admin users can exercise global oversight, while municipality-scoped reviewers remain restricted to their assigned scope.

8. Dashboards, Reporting & Maps

8.1 KPI Aggregation Layer

The reporting layer supports municipality-level and national oversight through aggregated queries for submission counts, workflow status distribution, municipality totals, and time-based trends. Aggregation remains RBAC-aware so that donor-facing views receive approved or anonymized slices, while internal administrative users can access the full scoped dataset.

8.2 Reporting posture

  • UNDP Admin: full cross-municipality visibility and export rights.
  • Municipality-scoped users: municipality-only worklists and dashboards.
  • Partner / Donor viewers: approved, aggregated, or anonymized reporting only.
  • Auditors: immutable activity review and governance-oriented reporting.

8.3 Geo-mapping support

  • Submissions and projects can carry latitude and longitude coordinates.
  • The backend provides map-ready endpoints for filtered datasets and marker display.
  • If no coordinate exists, no marker is rendered. The system fails gracefully rather than fabricating spatial data.
  • Mapping remains scope-aware; only authorized records are returned to the client.

10. Security & Privacy Controls

10.1 Transport and network security

  • TLS 1.2 or higher is required for all web, API, and object-storage transfers.
  • HSTS should be enabled at the web tier for production dashboard domains.
  • Secure cookie and secure token handling policies apply to first-party web access patterns.

10.2 Data protection

  • Minimal PII collection is preferred and access to sensitive identity fields is restricted to authorized internal roles.
  • Tokens are scoped and revocable.
  • Private evidence media is never exposed through public bucket policy.
  • Secrets remain server-side; no long-lived provider secrets should be stored inside the mobile binary.

10.3 Compliance readiness

  • Data minimization and purpose limitation guide the platform design.
  • Retention and archiving policy can be represented through system settings and storage lifecycle rules.
  • Audit logging supports accountability, incident review, and donor-grade traceability.
  • Role-scoped reporting prevents donor-facing visibility from exposing raw personal or unrestricted evidence data.

11. Performance & Scalability

11.1 Target operating posture

  • API p95 target: 500 ms or better for non-media endpoints.
  • Bulk synchronization is supported through queued background work and retry-safe API contracts.
  • Dashboard and reporting queries should be protected by proper indexes and pre-aggregated where necessary.

11.2 Scalability strategy

  • The API is logically stateless and therefore scale-ready beyond a single node.
  • Asynchronous work is separated from request-response paths through Laravel queues.
  • Object storage is decoupled from compute so evidence growth does not directly pressure VPS disk capacity.
  • The current database-queue deployment is suitable for the selected VPS footprint and has a clear upgrade path to dedicated queue infrastructure if throughput increases.

13. Hosting & Object Storage Plan (Libyan Spider VPS + AWS S3)

13.1 Compute hosting (Linux VPS)

The Laravel backend runs on a dedicated Linux VPS hosted by Libyan Spider as the primary compute node.

Selected VPS SpecificationValue
CPU2 vCPU
Memory8 GB RAM
Disk80 GB SSD
Bandwidth20 TB

The VPS hosts the web server, PHP-FPM, Laravel application, Vue dashboard, MySQL, scheduler, queue workers, runtime logs, and application configuration. This keeps core API logic, RBAC enforcement, validation workflows, and database transactions inside one controlled environment.

13.2 VPS storage scope

The VPS disk is reserved for application code, runtime configuration, MySQL data, logs, and static system assets shipped with the platform. User-uploaded evidence media and generated media variants are excluded from long-term VPS storage to prevent disk exhaustion and I/O contention.

13.3 Object storage for evidence media

All evidence media is stored in AWS S3 or an equivalent S3-compatible private object storage service. The database stores metadata only: object key, size, type, processing status, and variant references. This removes large binary I/O from the application server and allows storage capacity to scale independently from compute.

13.4 Secure media transfer via pre-signed URLs

  • Upload flow: the backend issues a time-limited pre-signed upload URL after validating authentication, scope, and media rules.
  • Download flow: the backend issues a time-limited download URL only after RBAC authorization and workflow visibility checks pass.
  • Buckets remain private. There is no public-read requirement.
  • This model reduces server bandwidth pressure because Laravel does not proxy large file transfers.

13.5 Evidence storage layout

The operational storage design should follow a clear object-key convention so evidence remains traceable to submission and media records.

evidence/raw/{submission_id}/{media_id}/original.ext
evidence/derived/{submission_id}/{media_id}/thumb.jpg
evidence/derived/{submission_id}/{media_id}/medium.jpg
evidence/derived/{submission_id}/{media_id}/large.jpg
evidence/video/{submission_id}/{media_id}/source.ext
evidence/video/{submission_id}/{media_id}/processed.mp4
evidence/video/{submission_id}/{media_id}/poster.jpg

Current implementation note: the repository already stores object metadata and derived variant paths. The exact prefix taxonomy can be standardized to the structure above without changing the overall architecture.

13.6 Access control and security controls

  • S3 buckets are private with public access blocked.
  • Pre-signed URLs are generated only by backend endpoints protected by middleware, policies, and scoped queries.
  • HTTPS protects both API traffic and storage transfer.
  • Object names should avoid leaking sensitive identifiers in user-visible filenames.
  • Expiring URLs limit the exposure window even after authorization is granted.

13.7 Operational notes

  • Queue workers handle variant generation triggers, video-processing triggers, export tasks, and push notification dispatch.
  • Monitoring should cover API health, queue backlog, failed jobs, database performance, OTP delivery errors, and S3 transfer errors.
  • This separation prevents long-running work from blocking API request latency and absorbs bursty sync traffic after offline periods.

14. Concluding Architecture Position

The UNDP Platform backend is structured as a secure, API-first Laravel system optimized for role-aware validation, offline-capable reporting, bilingual operation, and accountable evidence handling. The design keeps compute on the Libyan Spider VPS, protects evidence media in private object storage, and enforces access through a layered combination of authentication, RBAC middleware, policies, and query scoping.

This combination makes the platform suitable for municipal workflows, donor-safe reporting, and technical handover because the architecture is explicit, modular, auditable, and operationally scalable without changing its core security model.
