UNDP Platform Technical Interview Prep

Context used for this prep: Laravel 12 backend, Vue 3 frontend, Vite build, Pinia auth state, Axios API client, Leaflet maps, role and permission middleware, paginated project listing, project detail and edit modals, and backend validation for project data.

Best answer formula for "Why did you use... / Why did you not...":
1. State the reason.
2. State the tradeoff.
3. State the mitigation.
4. State the next improvement if needed.

Example: "I used server-side pagination to keep payloads small and response time stable. The tradeoff is extra API calls during navigation. That is acceptable here because it scales better than loading all rows at once. If the dataset grows further, I would add caching and query indexing."

Fast Opening Script

"This dashboard is API-driven with a clear frontend-backend split. Vue handles interactivity and user experience, while Laravel remains the source of truth for validation, authorization, access scoping, and auditability. The design choices prioritize maintainability, controlled data access, and safe scaling."

Core Principles to Repeat

  • The frontend improves usability, but the backend enforces security and data rules.
  • Permission-based UI improves clarity, but route middleware and access services are what actually protect the data.
  • We prefer predictable behavior over clever behavior: reload from the API after writes, clean up resources, and fail safely.
  • Resilience means graceful degradation, clear error handling, and safe defaults, not only uptime.

High-Probability "Why Did You Use..." Questions

Why did you use Laravel for the backend?
It gives us mature validation, routing, middleware, Eloquent relationships, and clear API structuring. That reduces boilerplate and makes authorization and audit logging easier to enforce consistently.
Why did you use Vue 3 for the dashboard?
The dashboard is interaction-heavy, so Vue gives reactive state, component isolation, and fast UI updates without overcomplicating the client. It is a good fit for forms, filters, and modal workflows.
Why did you use an API-driven frontend instead of rendering everything server-side?
It keeps the UI decoupled from backend logic, which improves maintainability and makes the backend reusable for future clients like mobile or external integrations.
Why did you use Axios with a shared API client?
A shared client centralizes the base URL, headers, and auth token handling. That keeps request behavior consistent and avoids duplicating auth logic across components.
Why did you use route middleware for permissions?
Because hiding buttons in the UI is not security. Route middleware ensures protected endpoints are blocked even if someone bypasses the frontend.
Why did you use server-side validation when the frontend already validates inputs?
Because the API is the trust boundary. Frontend validation improves user experience, but server-side validation protects the system from invalid or manipulated requests.
Why did you use pagination on the project list?
It keeps payload size controlled and prevents performance degradation as data grows. It is the safer default for scalability.
Why did you debounce search input?
To avoid firing a request on every keystroke. It reduces unnecessary load on the API and keeps the UI smoother under rapid input.
Why did you use Leaflet for the maps?
Leaflet is lightweight, reliable, and sufficient for marker-based project location selection and display. It fits the current use case without adding unnecessary platform complexity.
Why did you destroy and recreate map instances when modals close and reopen?
Because modal-based maps can leak memory or render incorrectly if old instances are reused carelessly. Explicit cleanup keeps behavior stable and prevents stale state.
Why did you reload data after create, update, or delete instead of trusting local state?
Reloading from the API ensures the UI reflects the real backend state. It is safer than assuming the local copy is fully accurate after a write.
Why did you include audit logging?
For accountability and traceability. In an administrative platform, being able to track who changed or deleted records is part of operational reliability.

High-Probability "Why Did You Not..." Questions

Why did you not load all projects at once?
That works only while the dataset is small. Pagination is more resilient because it keeps response times and memory usage predictable as the system grows.
Why did you not rely only on frontend permission checks?
Because frontend checks are only for user experience. Real protection must stay on the backend using middleware and access-scoped queries.
Why did you not keep a single long-lived map instance for the page?
The maps live inside modal lifecycles, so keeping a global instance adds more risk of stale DOM references and rendering issues. Local lifecycle cleanup is simpler and safer here.
Why did you not use optimistic UI updates?
For an admin dashboard, correctness matters more than perceived speed. I preferred confirming the backend result first to avoid showing users a state that might later fail validation.
Why did you not move everything into one giant global store?
Not all state benefits from being global. Local view state such as modal state, active filters, and temporary form values is easier to reason about when it stays close to the component.
Why did you not trust client-submitted reporter assignments directly?
Because assignments affect access and workflow. The backend re-validates that reporters exist, are active, have the correct role, and belong to the expected municipality.
Why did you not allow latitude without longitude, or vice versa?
A partial coordinate creates broken location data. The backend blocks that to preserve data integrity.
Why did you not allow end dates before start dates?
That would break timeline logic and reporting accuracy, so the backend explicitly validates chronological consistency.

Project-Specific Questions They May Target in This Screen

Why are the map elements stored as refs like detailMapEl and formMapEl?
Because Leaflet needs a real DOM node to mount into. Vue refs let the component safely initialize the map only after the element exists.
Why do you wait for the DOM before syncing the map?
Because the modal content may not be mounted yet. Waiting until the next render tick avoids initializing Leaflet against a missing or hidden container.
Why do you keep fallback option sets in the frontend if the backend can return options?
It improves resilience. If the options endpoint fails, the form can still remain usable with safe defaults instead of fully breaking.
Why do you filter assigned reporters again after municipality changes?
Because a municipality change changes which reporters are valid. Keeping invalid assignments would create inconsistent data and failed saves.
Why do you convert multiline text into arrays for fields like objectives and contacts?
It gives users a simple editing experience in the form while keeping the API payload structured and easier to validate.
Why do you use a confirmation dialog before deleting a project?
Because delete is destructive and may remove linked data. The confirmation is a simple safety barrier against accidental data loss.
Why do you keep loading, saving, and deleting states separately?
Different operations can fail or complete independently. Separate states make the UI more precise and prevent one action from masking another.
Why do you clear timers and remove map instances on component unmount?
That prevents memory leaks, stale listeners, and unexpected behavior if the user navigates away and returns later.

Tougher Follow-Up Questions and Strong Answers

Why are you storing the token in local storage instead of using an HTTP-only cookie?
The current implementation uses bearer tokens for simple SPA flow control. The tradeoff is that HTTP-only cookies are stronger against JavaScript access, so if the threat model becomes stricter, moving auth to secure cookies would be the stronger next step.
Why did you not add request cancellation for rapid filter changes?
Debouncing already reduces request volume, which was the first practical control. Adding cancellation would be a sensible next improvement to prevent stale responses winning under heavy interaction.
Why is the map logic still inside the component instead of a reusable composable?
The current scope is localized to one screen, so keeping it in the component keeps the implementation straightforward. If map behavior spreads to more screens, extracting a composable becomes the better maintainability move.
Why did you not add optimistic locking for concurrent edits yet?
The current flow is correct for normal usage, but true conflict prevention belongs in backend concurrency control. If simultaneous editing becomes common, I would add version or timestamp checks to stop silent overwrites.
Why did you not centralize all error handling in one system?
Shared handling exists at the API layer for auth failures, while view-level errors still stay local for precise user feedback. A more centralized notification pattern would be a good next refinement, but local handling keeps the current flows explicit.
Why did you not cache project lists aggressively?
Because correctness is more important than stale speed for administrative data. Caching should be added selectively, with invalidation rules, once the read volume clearly justifies it.

If They Push on Weaknesses, Use Honest Controlled Answers

  • On security: "The current pattern is acceptable for this app, but if security requirements tighten, I would prioritize HTTP-only cookie auth and a stricter CSP."
  • On performance: "The current dataset is handled safely with pagination and debounced search. If usage grows, the next steps are query indexing, request cancellation, and targeted caching."
  • On maintainability: "The screen is still readable as one view, but the next refactor would be extracting repeated modal or map logic into composables/components."
  • On concurrency: "Current behavior is functionally correct, but conflict resolution is a backend enhancement I would add if simultaneous edits become likely."
  • On UX: "The current confirmation flow is intentionally simple. If needed, it can be upgraded to a custom confirmation modal with clearer impact messaging."

Questions You Can Ask Back

  • Are you evaluating mainly for operational resilience, scaling readiness, or code maintainability?
  • Do you want the current implementation defended as-is, or are you also assessing the roadmap for production hardening?
  • Would you like me to explain the tradeoffs we accepted now versus what I would strengthen for a larger rollout?

What Not to Say

  • Do not say "the frontend blocks it, so users cannot do it."
  • Do not say "it works on my machine, so it is fine."
  • Do not say "we did not need validation because the form controls are restricted."
  • Do not pretend a missing production hardening feature already exists.
  • Do not defend every tradeoff as perfect. Defend it as appropriate for the current scope, then name the next upgrade path.

30-Second Closing Script

"The system is designed around safe defaults: backend-enforced authorization, validated API input, scoped data access, and predictable UI state. On the frontend, the choices favor maintainability and controlled interaction. On the backend, the priorities are correctness, access control, and auditability. Where there are future hardening opportunities, I can clearly identify them and explain the next step."
