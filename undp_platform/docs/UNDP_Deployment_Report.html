1. Overview

This deployment report is tailored to the UNDP Platform backend and should be used to govern production or pre-production releases of the actual system architecture in use. The platform is a secure, role-aware, API-first Laravel system serving web oversight users and mobile-connected field reporting workflows, with strict RBAC, offline-tolerant submission handling, bilingual Arabic / English responses, and immutable auditability.

The deployed platform consists of Laravel 12 running on a Libyan Spider VPS, MySQL as the primary transactional datastore, Eloquent as the ORM, Laravel queues using the database driver, Resala OTP for login delivery, Firebase Cloud Messaging for workflow notifications, and private S3-compatible object storage for evidence media and backup archives. User-uploaded evidence is intentionally excluded from VPS disk storage to protect compute performance and long-term storage stability.

2. Objective

This document records and governs deployment of the UNDP Platform according to its approved architecture. It is intended to show exactly what is being deployed, where it is hosted, which services are affected, how the release is executed, how security and RBAC are validated, how post-deployment testing is performed, and how rollback is handled if the release fails or introduces operational risk.

The deployment objective is to release code, configuration, schema changes, queue behavior, localization assets, object-storage integration, and notification services in a controlled and reversible manner while preserving availability, security, municipal data isolation, donor-safe visibility rules, and evidence integrity.

3. Architecture Summary

The deployment must preserve the backend architecture exactly as designed: API-first, permission-driven, municipality-aware, audit-heavy, and resilient to delayed mobile synchronization.

3.1 High-Level Architecture Principles

  • RESTful JSON API-first backend.
  • Strict RBAC enforcement at middleware, policy, and query-scoping layers.
  • Sanctum-based token authentication for API clients.
  • Offline-tolerant submission lifecycle with idempotent retry behavior through client UUIDs.
  • Immutable audit logging for governance-critical actions.
  • Private object storage for evidence media, separate from application compute.
  • Bilingual Arabic / English support at API and workflow level.

3.4 RBAC and Data Isolation Summary

The deployment must preserve the three-layer authorization model already implemented in the backend:

  • Middleware layer: request gatekeeping through auth:sanctum, active, permission:*, and any_permission:*.
  • Policy layer: business-rule enforcement through policies such as SubmissionPolicy.
  • Query scope layer: data isolation through scoped services such as SubmissionAccessService and ProjectAccessService.
Deployment implication: a successful release is not just “the site opens.” It must preserve municipality scope, ownership rules, donor-safe visibility, and active-user token revocation behavior.

5. Deployment Scope

In scope for this deployment:

  • Laravel backend release on the Libyan Spider VPS.
  • Frontend asset build and publication for the dashboard.
  • MySQL schema migrations and supporting data updates required by the release.
  • Sanctum auth flow validation, including OTP request, OTP verification, access tokens, and refresh tokens.
  • Queue worker restart and health verification for default, notifications, media, and exports.
  • Resala OTP and Firebase Cloud Messaging connectivity validation.
  • Private S3-compatible media storage validation, including presigned uploads and download URLs.
  • Localization, audit logging, reporting, and RBAC smoke testing.

Out of scope unless separately approved:

  • Mobile application binary deployment.
  • Hosting provider migration away from Libyan Spider.
  • Major operating system upgrade.
  • Changing the evidence object-key taxonomy as a standalone storage migration.
  • Replacing the current media-processing placeholder boundary with a new transcoding platform.
  • Any non-reversible schema or data migration not already approved for this release.

6. Deployment Steps

6.1 Enable maintenance mode

cd /var/www/undp-platform/current
php artisan down --render="errors::503" --retry=60

6.2 Create and store pre-deployment database backup

mkdir -p /var/backups/undp-platform
mysqldump --single-transaction --routines --triggers \
  -h [db-host] -u [db-user] -p'[db-password]' [db-name] \
  > /var/backups/undp-platform/predeploy_[release_id].sql

If backup policy requires off-node retention, upload the dump to the approved S3-compatible backup bucket after creation.

6.3 Prepare the release directory

export RELEASE_ID=[YYYYMMDD_HHMMSS]
export APP_ROOT=/var/www/undp-platform
export RELEASE_PATH="$APP_ROOT/releases/$RELEASE_ID"

mkdir -p "$APP_ROOT/releases" "$APP_ROOT/shared"
git clone --branch [release-tag-or-branch] --depth 1 [repository-url] "$RELEASE_PATH"
cd "$RELEASE_PATH"

cp "$APP_ROOT/shared/.env" .env
ln -sfn "$APP_ROOT/shared/storage" storage

6.4 Install dependencies and build assets

cd "$RELEASE_PATH"
composer install --no-dev --prefer-dist --optimize-autoloader
npm ci
npm run build

6.5 Validate deployment-critical environment settings

cd "$RELEASE_PATH"
php artisan about
php artisan tinker --execute="dump([
  'app_env' => config('app.env'),
  'queue' => config('queue.default'),
  'media_disk' => config('media.disk'),
  'media_direct_upload_disk' => config('media.direct_upload_disk'),
  'otp_expires_in' => config('otp.expires_in_seconds'),
  'otp_cooldown' => config('otp.resend_cooldown_seconds'),
  'resala_enabled' => config('services.resala.enabled'),
]);"

6.6 Run Laravel optimization and migrations

cd "$RELEASE_PATH"
php artisan optimize:clear
php artisan migrate --force
php artisan storage:link || true
php artisan config:cache
php artisan route:cache
php artisan view:cache
php artisan event:cache

6.7 Switch the live symlink

ln -sfn "$RELEASE_PATH" "$APP_ROOT/current"
cd "$APP_ROOT/current"

6.8 Restart queues and runtime services

cd "$APP_ROOT/current"
php artisan queue:restart

sudo supervisorctl reread
sudo supervisorctl update
sudo supervisorctl restart undp-platform-worker:*

sudo systemctl reload php8.2-fpm
sudo systemctl reload nginx

If workers are split by queue name instead of a single worker group, ensure separate coverage for default, notifications, media, and exports.

6.9 Confirm scheduler and reopen traffic

crontab -l | grep "schedule:run"
php artisan up

6.10 Example queue worker commands for this architecture

php artisan queue:work database --queue=default --sleep=3 --tries=3 --timeout=120
php artisan queue:work database --queue=notifications --sleep=3 --tries=3 --timeout=120
php artisan queue:work database --queue=media --sleep=3 --tries=3 --timeout=600
php artisan queue:work database --queue=exports --sleep=3 --tries=3 --timeout=300

7. Database Notes

The deployed backend is MySQL-centric. Eloquent models, workflows, audit logs, and queue tables all depend on schema integrity and correct charset configuration.

7.1 Key operational tables affected by deployment

Table / AreaPurpose
usersRoles, status, municipality assignment, preferred locale, and device-token context.
otp_codesOTP issuance, expiry, and verification-attempt tracking.
submissionsPrimary reporting records, including client_uuid idempotency support.
submission_status_eventsImmutable workflow timeline for status transitions.
media_assetsObject-storage metadata, checksums, statuses, and variant references.
audit_logsAppend-only audit trail for sensitive actions.
workflow_statuses / validation_reasonsConfigurable workflow dictionaries.
jobs, failed_jobs, job_batchesDatabase queue transport and failed-job tracking.
system_settingsPersistent settings affecting workflow, localization, security, and reporting.

7.2 Deployment considerations

  • Use utf8mb4 or equivalent full Unicode support so Arabic and English content remains safe across submissions, labels, exports, and audit data.
  • Treat migrations as release-critical because both workflow and queue behavior depend on schema correctness.
  • Because the queue driver is database-backed, queue table health is part of database readiness.
  • Do not assume migrate:rollback is the safest recovery path for production. For irreversible or risky changes, restoring the verified pre-deployment backup may be the correct rollback action.
  • Audit log integrity must be preserved across restore operations; never “clean up” or mutate audit entries as part of deployment troubleshooting.

8. S3 Media Storage Notes

User evidence media and backup archives are stored in AWS S3 or an equivalent S3-compatible private object storage service. The application is not hosted on S3. The application is hosted on the Libyan Spider VPS; S3 is used only for storage operations.

8.1 Expected production settings

8.3 Current implementation notes

  • The current backend already supports authenticated presigned uploads when the media disk driver is S3.
  • Upload completion dispatches a media-processing job onto the media queue.
  • The current application stores media object keys and derived variants in the database and generates temporary download URLs through the storage layer.
  • The current object-key pattern is application-managed. If the platform later standardizes to a stricter evidence prefix layout such as evidence/raw and evidence/derived, that change should be handled as a planned storage migration, not as a routine deployment side-effect.

8.4 Storage validation points

  • The bucket remains private and no public-read access is enabled.
  • Presigned upload URLs are issued only after authenticated and authorized requests.
  • Download URLs expire correctly and are never handed to unauthorized roles.
  • Donor or partner-facing views do not expose unrestricted raw media access.
  • Object-storage latency or signature errors do not block the entire application from starting, but they must fail visibly and be monitored.

9. Rollback Plan

  1. Return the application to maintenance mode if user-facing impact or data-risk is confirmed.
  2. Switch the live symlink back to the last verified stable release.
  3. Clear and rebuild Laravel caches on the restored release.
  4. Restart queue workers, reload PHP-FPM, and reload the web server.
  5. Evaluate whether database restoration is required. If the release included irreversible schema or data changes, restore the pre-deployment MySQL backup.
  6. Validate login, RBAC scoping, object-storage access, audit logging, and queue health before reopening access.

10. Risks and Mitigation

Resala OTP outage or rejectionUsers cannot authenticate and the platform becomes effectively unavailable.Validate provider connectivity before deployment, confirm credentials, and monitor OTP failures immediately after release.
Database queue backlogDelayed media processing, exports, and notifications, especially after offline reconnect bursts.Review queue depth before deployment, ensure worker coverage by queue, and monitor failed jobs and backlog after release.

11. Recommendations

  • Retain a symlink-based release model on the Libyan Spider VPS to keep rollback fast and controlled.
  • Keep queue workers separated operationally by queue name or at minimum monitor worker coverage for default, notifications, media, and exports.
  • Automate off-node backup upload and periodic restore drills for MySQL and storage-backed archives.
  • Promote queue backlog, OTP failures, FCM failures, object-storage errors, and audit-write failures to first-class release monitoring signals.
  • Keep production media storage private and preserve short-lived signed URL behavior for all evidence access.
  • Document any role or permission changes as deployment-sensitive changes because they directly affect municipality scope and donor-safe visibility.
  • Use the existing asynchronous media-processing boundary to introduce a full image/video pipeline when production demand requires it, rather than blocking API requests with inline processing.
