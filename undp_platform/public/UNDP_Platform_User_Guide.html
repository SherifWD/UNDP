1. Guide Scope and Navigation

This guide covers the currently implemented web dashboard, validation screens, reporting screens, user administration, settings, and the API surface used by the mobile application. It is written against the live codebase and the current seeded dataset.

The cover page works as a clickable table of contents. In the HTML guide, each link jumps directly to the relevant section. In the Word guide, the same numbering and internal bookmarks are preserved.

All walkthroughs below reflect the current implementation. Where a full CRUD action is not exposed in the web UI, the guide states that explicitly instead of implying functionality that does not exist.

5. Authentication Module

Authentication is phone-number based and uses OTP verification. The same token is reused by the web app and can be used in Postman/mobile during development.

5.1 Login Page (/login)

Purpose: Start the OTP request flow.

Main buttons and controls

  • Country code selector
  • Phone number field
  • Continue

Operational steps

  1. Open the login page.
  2. Select the country code if needed. The default local setup uses +218.
  3. Enter the phone number and click Continue.
  4. If validation passes, the app navigates to the OTP page.
  5. If validation fails, the field shows an inline error and the request is blocked.
Back to Contents

5.2 OTP Verification (/otp)

Purpose: Complete the login and create the authenticated session.

Main buttons and controls

  • OTP input fields
  • Verify
  • Resend
  • Change Number

Operational steps

  1. Enter the 6-digit OTP from the configured SMS / log provider.
  2. Click Verify.
  3. On success, the app stores the token and user object locally, then redirects to the dashboard.
  4. Use Resend if the code expires or is not received.
  5. Use Change Number to return to the login page and restart the flow.
Back to Contents

5.3 Logout and Session Control

Purpose: Terminate the session safely from the sidebar footer card.

Operational steps

  1. Locate the logged-in user card at the bottom of the sidebar.
  2. Click the logout action on the right side of the card.
  3. The stored token is removed and the app returns to the login page.

6. Dashboard Module

The dashboard combines KPIs with a municipality and project workspace. This is the main landing page for most authenticated users.

6.1 KPI Strip and Global Overview

Purpose: Review headline metrics before drilling into projects.

Main buttons and controls

  • All Projects dropdown
  • All Sources dropdown
  • All Time dropdown
  • Add New Project

Operational steps

  1. Open the dashboard after login.
  2. Review the Reports donut, Funding Progress block, and Beneficiaries Overview block.
  3. Use the top-level dropdowns to scope visible metrics where available.
  4. Use Add New Project to jump into project creation if your role has permission.
Back to Contents

6.2 Municipality and Project Workspace

Purpose: Browse projects by municipality and map them spatially.

Main buttons and controls

  • Municipality dropdown
  • Search projects
  • Filter
  • Project list cards

Operational steps

  1. Use the Municipality dropdown to scope the map and project list. Leaving it on All Municipalities keeps the full scoped dataset visible.
  2. Use Search projects to filter the visible project rail by name or reference.
  3. Review the map markers and the live project count in the right rail.
  4. Select a project card to drill into the detail pane.
Back to Contents

6.3 Filter Panel and Drill-Down

Purpose: Refine the project rail using priority, area, and status.

Main buttons and controls

  • Filter
  • Apply
  • Reset
  • Go to Submission
  • Close details

Operational steps

  1. Click Filter next to the search field.
  2. Choose priority, area, and status as needed.
  3. Click Apply to update the project rail and map focus.
  4. Click Reset to clear only the dashboard-side project filters.
  5. After opening a project, use Go to Submission to move directly into the project-specific submission page.

7. Municipality Management

Municipalities are currently managed from the Projects module and then reused throughout filters, tabs, and dashboards.

7.1 Create a Municipality

Purpose: Add a municipality record used by projects and scoped dashboards.

Operational steps

  1. Open Projects.
  2. Click the Create Municipality button in the page header.
  3. Enter English name, Arabic name, and the municipality code.
  4. Click Save Municipality to persist the new record.
  5. The municipality becomes available in project forms, tabs, and dashboard selectors after reload.
Back to Contents

7.2 Read and Use Municipalities

Municipality update exists at the API layer, but the current web UI does not expose a dedicated municipality edit screen. Hard delete is not implemented.

Purpose: Use municipality records to segment data.

Operational steps

  1. In Projects, municipality tabs split the project list by municipality.
  2. In Dashboard and Reports & Map, municipality selectors scope the visible markers and project lists.
  3. In Settings and User Management, municipality assignment determines what some roles can review.

8. Project Management

Projects are the core container for reporting. The web UI supports create, list, detail, and edit flows.

8.1 Browse the Project Registry

Purpose: Read project rows and key metrics.

Main buttons and controls

  • Search
  • Status filter
  • Municipality tabs
  • View Submission
  • Edit

Operational steps

  1. Open Projects from the sidebar.
  2. Use search and status filter to narrow the rows.
  3. Use municipality tabs to limit the table to one municipality.
  4. Review approved submissions, pending submissions, progress, and current status directly in the table.
Back to Contents

8.2 Create a Project

Purpose: Create a new project record.

Operational steps

  1. Click Add New Project in the Projects header.
  2. Select the municipality and enter English and Arabic names.
  3. Enter a description, choose status, and set latitude/longitude.
  4. Click Save Project.
  5. After save, verify that the new row appears in the registry and on the dashboard map.
Back to Contents

8.3 View Project Details

Purpose: Open the project detail modal for a full read-only summary.

Operational steps

  1. Click any project row in the Projects table.
  2. Review the project status, municipality, last update, location, description, progress, and submission stats.
  3. Use Go to Submissions to jump to the project-specific submission page.
Back to Contents

8.4 Edit a Project

Purpose: Update project metadata.

Operational steps

  1. Click Edit in a project row, or open the detail modal and click Edit there.
  2. Modify any allowed field: municipality, names, description, status, latitude, or longitude.
  3. Click Update Project to save the change.
  4. Re-open the detail modal to confirm the saved values.
Back to Contents

8.5 Project Deletion and Archiving Note

Purpose: Clarify current limitations.

Operational steps

  1. The current implementation does not expose a project delete action.
  2. Use the status field to switch between active and archived where appropriate.
  3. If a true deletion policy is needed, it must be added deliberately with audit controls first.

9. Submission Management

Submission records are reviewed in the web UI. Submission creation itself belongs to the mobile or direct API flow.

9.1 Project Submission List

Purpose: Inspect submission counts and row-level records for one project.

Main buttons and controls

  • Back
  • Status filter
  • Search submissions
  • Export CSV
  • Export PDF
  • View Details

Operational steps

  1. From Projects, click View Submission for the target project.
  2. Review the KPI strip for total submissions, pending actions, and evidence coverage.
  3. Use the table to inspect reporter name, report type, region, attachments, and status.
  4. Use export buttons to generate the current view in CSV or PDF.
Back to Contents

9.2 Open Submission Detail

Purpose: Read the complete submission, media assets, and timeline.

Operational steps

  1. Click View Details on any submission row.
  2. Review the header details: status, project, municipality, reporter, description, and validation comment.
  3. Inspect media evidence and click any asset to load a preview.
  4. Review the immutable timeline at the bottom of the page.
Back to Contents

9.3 Approve, Rework, and Reject

Purpose: Process a submission through the validation workflow.

Operational steps

  1. Open a submission detail page.
  2. Choose Approve, Request Rework, or Reject.
  3. For rework and rejection, provide a comment. Use a reason template where available.
  4. Click the confirm button in the action modal.
  5. The status updates and the timeline is extended with the new event.
Back to Contents

9.4 Submission Creation Path

This separation is intentional: the web app is currently focused on review, oversight, administration, and reporting, while field submission entry belongs to mobile/API flows.

Purpose: Clarify where new submissions are created.

Operational steps

  1. The current admin web UI does not expose a Create Submission form.
  2. New submissions are expected to be created by the mobile application or through direct API integration.
  3. Use the Postman collection in this repository as the reference for the Flutter/mobile integration team.

10. Validation Worklist

The validation worklist is the queue used by validators and municipal focal points to process pending items efficiently.

Main buttons and controls

  • Search
  • Sort
  • Project filter
  • Status filter
  • Date filters
  • Apply
  • Reset

Operational steps

  1. Open Submissions from the sidebar (the route maps to the validation worklist).
  2. Filter the queue by project, status, or date as needed.
  3. Open the target item to move into the Submission Detail page.

12. User Management

User Management covers onboarding, role assignment, municipality assignment, and account status control.

12.1 User Overview and Read

Purpose: Review user distribution and the current user registry.

Operational steps

  1. Open User Management from the sidebar.
  2. Review the reporter donut, municipality distribution, and user classification bar.
  3. Use the table to inspect name, email, phone, role, and current status actions.

12.2 Add a New User

Purpose: Create a new user account and assign the base role.

Operational steps

  1. Click Add New User.
  2. Enter full name, email, country code, and phone number.
  3. Choose the user role, organization, and assigned municipality if applicable.
  4. Click Add to create the account.
  5. The new user appears in the table after the list reloads.

12.3 Update Role, Municipality, and Permission Profile

Purpose: Edit an existing user without recreating the account.

Operational steps

  1. Click Set Permissions on the target row.
  2. Update the username, email, phone, role, or municipality assignment.
  3. Use the permission preview grid to confirm the effective role-based permission set.
  4. Click Save Changes.
Back to Contents

12.4 Disable or Re-enable a User

Hard delete is not exposed in the current implementation. Disabling the account is the supported control path.

Purpose: Control access without deleting the account.

Operational steps

  1. Use the Disable button on the target row to suspend access.
  2. Provide a reason if desired and click Confirm.
  3. To restore access later, use the Enable button on the same row.
  4. This action is logged to the audit trail.

13. Audit Log

Audit logs are immutable records of key actions across the platform. They support filtering, review, and forensic traceability.

13.1 Audit Table and Filters

Purpose: Review high-level activity and find target events.

Operational steps

  1. Open Audit Log from the sidebar.
  2. Use action, user ID, and date filters to narrow the dataset.
  3. Use Today or Last 7 Days for quick filters.
  4. Click Apply to refresh the table, or Reset to clear the filter set.
Back to Contents

13.2 Open Audit Entry Detail

Audit records are view-only and intentionally immutable. There is no create, edit, or delete action in this module.

Purpose: Inspect the before, after, and metadata payload for one event.

Operational steps

  1. Click any audit row or its reference link.
  2. Review the Before payload, After payload, and Metadata block.
  3. Use this view when confirming who changed what and when.

14. Settings Module

The Settings page is now backed by the persistent system settings API. It supports save and reset behavior by tab for authorized users.

14.1 General Tab

Purpose: Maintain organization profile, localization defaults, and system defaults.

Operational steps

  1. Open Settings and stay on the General tab.
  2. Update organization profile fields, localization defaults, and reporting defaults as needed.
  3. Use the Add/Edit/Delete mini-controls in status and risk option sets where shown.
  4. Click Save Changes to persist the active tab.
Back to Contents

14.2 Users & Roles Tab

Purpose: Review role definitions and the permission matrix.

Operational steps

  1. Click the Users & Roles tab.
  2. Review the role definition blocks and the core permissions table.
  3. Confirm the permission matrix before changing user roles in User Management.
  4. Save changes if you adjust any editable control in this tab.
Back to Contents

14.3 Reporting & Workflow Tab

Purpose: Configure workflow mode, deadlines, escalation, evidence rules, and notification behavior.

Operational steps

  1. Click the Reporting & Workflow tab.
  2. Set workflow mode, default submission status, approval requirements, escalation rules, and evidence requirements.
  3. Adjust email and in-app notification toggles as required by policy.
  4. Click Save Changes after completing the workflow configuration.
Back to Contents

14.4 Security Tab

Purpose: Manage security posture for authentication and audit controls.

Operational steps

  1. Click the Security tab.
  2. Review or update two-factor authentication, SSO, password rules, RBAC toggle, IP restrictions, and audit toggles.
  3. Use Save Changes to persist the security configuration.
Back to Contents

14.5 Save and Reset Behavior

Purpose: Use settings actions safely.

Operational steps

  1. Each tab can be edited independently.
  2. Save Changes writes the currently active tab payload to the backend settings record.
  3. Reset restores the visible tab values from the last saved backend state.
  4. Users without workflow.manage can still view settings, but cannot persist changes.

16. Mobile API Quick Start

The mobile application should use the Postman collection in docs/postman as the starting contract. The current web UI already proves the OTP auth, project lookup, submission review, media, and settings endpoints.

Recommended first mobile flow: request OTP, verify OTP, call /auth/me, load municipalities, load scoped projects, create submissions via the API, upload media via presign/complete, then poll the user submission list for status changes.

  • Collection: docs/postman/UNDP_Mobile_API.postman_collection.json
  • Environment: docs/postman/UNDP_Mobile_API_Local.postman_environment.json
  • Base URL (local): http://127.0.0.1:8000/api
